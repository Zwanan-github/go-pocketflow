## Context

go-pocketflow 是一个极简的 LLM 应用框架，目前只支持同步执行模式。现有架构基于 Node 和 Flow 两个核心概念，通过链式调用构建工作流。随着应用场景的扩展，对并发处理和非阻塞执行的需求日益增长，特别是在处理多个 LLM 调用和 I/O 密集型任务时。

**当前限制：**
- 所有执行都是同步的，无法处理并发请求
- BatchFlow 的批量处理是串行的，性能受限
- 缺乏超时和取消机制，不适合长时间运行的任务
- 没有错误恢复和重试机制

**技术约束：**
- 必须保持向后兼容性，现有 API 不能破坏
- 遵循 Go 的并发哲学，使用 goroutines 和 channels
- 保持框架的极简设计理念
- 避免引入复杂的外部依赖

## Goals / Non-Goals

**Goals:**
- 提供高性能的异步执行能力，支持并发处理多个节点
- 实现灵活的并发控制，包括并发限制和超时管理
- 构建健壮的错误处理机制，支持错误传播和重试
- 保持 API 的一致性和易用性，与现有同步 API 形成互补
- 优化批量处理性能，支持真正的并发批量执行

**Non-Goals:**
- 不实现分布式执行或跨节点协调
- 不引入消息队列或外部协调系统
- 不改变现有的同步执行语义
- 不实现复杂的流式处理（为未来扩展预留空间）

## Decisions

### 1. 异步模型选择：Channel-based vs Callback-based
**决策：** 使用 Channel-based 模型
**理由：**
- 符合 Go 的惯用法，channels 是 Go 并发的核心
- 提供更好的组合性和可读性
- 便于实现取消和超时机制
- 与现有函数式风格保持一致

**替代方案考虑：** Callback-based 模式被拒绝，因为会导致回调地狱，降低代码可读性。

### 2. 并发控制实现：Semaphore vs Worker Pool
**决策：** 使用 Semaphore 模式
**理由：**
- 更灵活，支持动态调整并发数
- 实现简单，资源开销小
- 与 goroutine 生命周期管理配合良好
- 支持优雅的取消机制

**替代方案考虑：** Worker Pool 被拒绝，因为过于重量级，不适合框架的轻量级特性。

### 3. 状态管理：Sync.Map vs RWMutex + Map
**决策：** 使用 RWMutex + Map
**理由：**
- 性能更好，特别是在读多写少的场景
- 更细粒度的控制，便于实现原子操作
- 与现有 SharedStore (map[string]any) 保持一致
- 避免引入新的抽象概念

**替代方案考虑：** Sync.Map 被拒绝，因为类型限制和性能考虑。

### 4. 错误处理：Error Channel vs Context Error
**决策：** 使用 Error Channel
**理由：**
- 专门用于异步错误传播
- 不与 context cancellation 混淆
- 支持多种错误类型和聚合
- 便于实现错误恢复和重试

### 5. API 设计：New Types vs Extension Methods
**决策：** 创建新的 AsyncNode 和 AsyncFlow 类型
**理由：**
- 明确区分同步和异步语义
- 避免现有 API 的复杂性
- 便于后续功能扩展
- 保持向后兼容性

## Risks / Trade-offs

### 1. Goroutine 泄漏风险
**风险：** 异步执行可能导致 goroutine 泄漏
**缓解措施：**
- 使用 context.Context 进行生命周期管理
- 实现 defer 模式确保资源清理
- 提供显式的取消机制
- 添加 runtime.GC() 友好的设计

### 2. 内存使用增加
**风险：** 并发执行会显著增加内存使用
**缓解措施：**
- 实现可配置的并发限制
- 使用对象池复用临时对象
- 提供流式处理选项
- 添加内存使用监控

### 3. 调试复杂性
**风险：** 异步代码更难调试和理解
**缓解措施：**
- 提供详细的日志记录机制
- 实现执行追踪功能
- 添加调试模式和可视化工具
- 保持清晰的错误信息

### 4. 向后兼容性维护
**风险：** 新功能可能影响现有代码
**缓解措施：**
- 严格的接口分离，同步和异步互不干扰
- 全面的回归测试
- 渐进式迁移路径
- 详细的迁移文档
